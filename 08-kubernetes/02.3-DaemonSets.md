# DaemonSets

This is a very important Kubernetes concept because **DaemonSets are used heavily in real production clusters**, especially by DevOps and SRE teams.

A **DaemonSet** ensures that **one copy of a pod runs on every node** (or selected nodes) in a Kubernetes cluster.

Think:

> Deployment → "Run 5 pods anywhere in the cluster."
> 

> DaemonSet → "Run 1 pod on every node."
> 

---

## Real-world analogy

Imagine you're responsible for collecting security camera footage from every office building.

You need:

```
Building 1 → Agent
Building 2 → Agent
Building 3 → Agent
Building 4 → Agent
```

Not:

```
Building 1 → Agent
Building 2 → No Agent
Building 3 → No Agent
Building 4 → 3 Agents
```

A DaemonSet guarantees exactly that.

---

## Why do we need DaemonSets?

Some applications must run on **every node** because they collect or provide node-level information.

Examples:

| Use Case | Why Every Node Needs It |
| --- | --- |
| Log Collection | Collect logs from every node |
| Monitoring | Gather CPU, memory, disk metrics |
| Security Agents | Scan every node |
| Networking Plugins | Manage networking on every node |
| Storage Plugins | Mount storage on every node |

## Deployment vs DaemonSet

### Deployment

```yaml
replicas: 3
```

Cluster:

```
Node-1 → App
Node-2 → App
Node-3 → App
```

But Kubernetes may place:

```
Node-1 → 2 Pods
Node-2 → 1 Pod
Node-3 → 0 Pods
```

That's okay for a Deployment.

---

### DaemonSet

Cluster:

```
Node-1 → 1 Pod
Node-2 → 1 Pod
Node-3 → 1 Pod
```

Always.

Add a new node:

```
Node-4 added
```

Kubernetes automatically creates:

```
Node-4 → 1 Pod
```

No manual action required.

---

## What happens when a new node joins?

Suppose cluster grows:

```
Before:
Node-1
Node-2
Node-3
```

DaemonSet:

```
3 Pods
```

Now:

```
Node-4 added
```

Kubernetes automatically schedules:

```
Node-4 → DaemonSet Pod
```

This is the biggest feature of DaemonSets.

---

## YAML Example

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: fluentd
spec:
  selector:
    matchLabels:
      app: fluentd
  template:
    metadata:
      labels:
        app: fluentd
    spec:
      containers:
      - name: fluentd
        image: fluent/fluentd
```

Apply:

```bash
kubectl apply -f daemonset.yaml
```

---

## How to check DaemonSets?

```bash
kubectl get daemonsets
```

Output:

```
NAME       DESIRED CURRENT READY
fluentd    3       3       3
```

Meaning:

```
3 Nodes
3 Pods running
```

## Interview Scenario

**Interviewer:**

"Why would you use a DaemonSet instead of a Deployment for Fluentd?"

Good answer:

> "Fluentd needs access to logs on every node. A Deployment only guarantees a certain number of replicas, but not one pod per node. A DaemonSet ensures that every current and future node automatically runs a Fluentd pod, allowing complete log collection across the cluster."
> 

That's exactly how an SRE would explain it.

## Quick Comparison Table

| Feature | Deployment | DaemonSet |
| --- | --- | --- |
| Fixed replicas | ✅ | ❌ |
| One pod per node | ❌ | ✅ |
| Auto-run on new nodes | ❌ | ✅ |
| Used for applications | ✅ | Sometimes |
| Used for monitoring/logging agents | ❌ | ✅ |
