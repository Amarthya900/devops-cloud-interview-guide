# OSI Layers, Load Balancers, Ingress, and Why We Need Both

This is one of the most important networking topics for Kubernetes, DevOps, and SRE interviews.

---

# 1. Why Do We Have Layers?

Networking is divided into layers so that each layer has a specific responsibility.

Think:

```text
Application
   ↓
Transport
   ↓
Network
   ↓
Physical Network
```

Instead of memorizing all 7 OSI layers initially, focus on the 3 layers DevOps engineers use daily.

| Layer   | Name        | Examples         | DevOps Relevance         |
| ------- | ----------- | ---------------- | ------------------------ |
| Layer 7 | Application | HTTP, HTTPS, DNS | Ingress, API Gateway     |
| Layer 4 | Transport   | TCP, UDP         | Load Balancers, Services |
| Layer 3 | Network     | IP Routing       | Pod Networking, CNI      |

---

# 2. Layer 3 (Network Layer)

Responsible for:

```text
IP Addresses
Routing
Packet Delivery
```

Example:

```text
Pod A
10.244.1.10

Pod B
10.244.2.20
```

Question:

```text
How does packet reach Pod B?
```

Answer:

```text
Layer 3 Routing
```

---

## DevOps Examples

### Check IP Routes

```bash
ip route
```

Example:

```text
10.244.2.0/24 via worker-2
```

Meaning:

```text
Traffic for 10.244.2.x
go to worker-2
```

---

# 3. Layer 4 (Transport Layer)

Responsible for:

```text
TCP
UDP
Ports
Load Balancing
```

Example:

```text
10.244.1.10:80
```

IP:

```text
10.244.1.10
```

Port:

```text
80
```

Layer 4 uses both.

---

## Kubernetes Service is Layer 4

Example:

```yaml
kind: Service
spec:
  ports:
  - port: 80
```

Service doesn't care about:

```text
/api
/login
/admin
```

It only cares about:

```text
TCP 80
TCP 443
UDP 53
```

This is Layer 4 routing.

---

## Layer 4 Load Balancer

Example:

```text
Client
   ↓
LB
   ↓
Pod1
Pod2
Pod3
```

LB only sees:

```text
TCP Port 80
```

It does NOT understand URLs.

---

# 4. Layer 7 (Application Layer)

Responsible for:

```text
HTTP
HTTPS
DNS
Headers
Cookies
URLs
```

Example Request:

```http
GET /api/users
```

Only Layer 7 devices understand this.

---

# Ingress is Layer 7

Ingress can inspect:

```text
Host Header
Path
HTTP Method
```

Example:

```text
app.company.com
```

and

```text
/api
```

---

## Example

```text
Request:

app.company.com/api
```

Ingress:

```text
Route to API Service
```

Request:

```text
app.company.com/web
```

Ingress:

```text
Route to Web Service
```

Layer 4 cannot do this.

---

# Layer 4 vs Layer 7

| Feature             | Layer 4 | Layer 7 |
| ------------------- | ------- | ------- |
| Understand TCP/UDP  | ✅       | ✅       |
| Understand HTTP     | ❌       | ✅       |
| Understand URL Path | ❌       | ✅       |
| Understand Hostname | ❌       | ✅       |
| Route by `/api`     | ❌       | ✅       |
| Route by domain     | ❌       | ✅       |

# Option 1: LoadBalancer Service

```yaml
apiVersion: v1
kind: Service

spec:
  type: LoadBalancer
```

Architecture:

```text
Internet
   ↓
Cloud Load Balancer
   ↓
Service
   ↓
Pods
```

---

## Limitation

Suppose:

```text
App1
App2
App3
```

You create:

```text
LB for App1
LB for App2
LB for App3
```

Result:

```text
3 Load Balancers
3 Public IPs
```

Expensive.

---

# Option 2: Ingress

Architecture:

```text
Internet
   ↓
Load Balancer
   ↓
Ingress Controller
   ↓
Services
   ↓
Pods
```

---

# Example Ingress

Suppose:

```text
api.company.com
```

and

```text
shop.company.com
```

---

Ingress YAML:

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress

metadata:
  name: app-ingress

spec:
  rules:

  - host: api.company.com

    http:
      paths:
      - path: /
        pathType: Prefix

        backend:
          service:
            name: api-service
            port:
              number: 80

  - host: shop.company.com

    http:
      paths:
      - path: /
        pathType: Prefix

        backend:
          service:
            name: shop-service
            port:
              number: 80
```

---

# Why Can't We Use Only Ingress?

This is the most common interview question.

---

## Problem

Ingress is just a Kubernetes object.

Example:

```yaml
kind: Ingress
```

This YAML itself cannot receive internet traffic.

---

You need an actual component:

```text
NGINX Ingress Controller
Traefik
HAProxy
```

that listens for traffic.

---

# How Traffic Reaches Ingress?

You expose the Ingress Controller using a LoadBalancer Service.

Example:

```text
Internet
   ↓
Load Balancer
   ↓
Ingress Controller
```

Without LB:

```text
Internet
      X
Ingress Controller
```

No public entry point.

---

# Real Production Architecture

```text
Internet
   ↓

Cloud Load Balancer
(AWS ELB / Azure LB / GCP LB)

   ↓

Ingress Controller
(NGINX)

   ↓

Ingress Rules

   ↓

Services

   ↓

Pods
```


# One-Line Memory Trick

```text
Layer 3 → Where should the packet go? (IP)

Layer 4 → Which application port? (TCP/UDP)

Layer 7 → Which website, host, or URL? (HTTP/HTTPS)

LoadBalancer gives external connectivity.

Ingress provides smart HTTP/HTTPS routing.

Ingress usually sits behind a LoadBalancer.
```

## Summary Table

| Question                          | Answer                                                       |
| --------------------------------- | ------------------------------------------------------------ |
| What is Layer 3?                  | IP routing between networks                                  |
| What is Layer 4?                  | TCP/UDP and port-based traffic handling                      |
| What is Layer 7?                  | HTTP/HTTPS-aware routing                                     |
| Is Service Layer 4?               | Yes                                                          |
| Is Ingress Layer 7?               | Yes                                                          |
| Can Ingress replace LoadBalancer? | No                                                           |
| Why not?                          | Ingress needs an externally reachable entry point            |
| What provides that entry point?   | A LoadBalancer (or NodePort) exposing the Ingress Controller |
