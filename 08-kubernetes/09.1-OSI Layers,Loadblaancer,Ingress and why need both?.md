# OSI Layers, Load Balancers, Ingress, and Why We Need Both

This is one of the most important networking topics for Kubernetes, DevOps, and SRE interviews.

---

# 1. Why Do We Have Layers?

Networking is divided into layers so that each layer has a specific responsibility.

Think:

```
Application
   ↓
Transport
   ↓
Network
   ↓
Physical Network
```

Instead of memorizing all 7 OSI layers initially, focus on the 3 layers DevOps engineers use daily.

| Layer | Name | Examples | DevOps Relevance |
| --- | --- | --- | --- |
| Layer 7 | Application | HTTP, HTTPS, DNS | Ingress, API Gateway |
| Layer 4 | Transport | TCP, UDP | Load Balancers, Services |
| Layer 3 | Network | IP Routing | Pod Networking, CNI |

In addition to TCP and UDP, Layer 4 of the OSI model includes several other protocols, each designed for specific use cases and requirements:

- **SCTP**: Reliable, supports multiple streams, used in telecommunications.
- **DCCP**: Connection-oriented, focuses on congestion control, used in streaming and gaming.
- **RUDP**: Combines reliability with low overhead, suitable for real-time communications.
- **QUIC**: Built on UDP, aims for low latency and enhanced performance for web applications.

# 2. Layer 3 (Network Layer)

Responsible for:

```
IP Addresses
Routing
Packet Delivery
```

Example:

```
Pod A
10.244.1.10

Pod B
10.244.2.20
```

Question:

```
How does packet reach Pod B?
```

Answer:

```
Layer 3 Routing
```

# 3. Layer 4 (Transport Layer)

Responsible for:

```
TCP
UDP
Ports
Load Balancing
```

Example:

```
10.244.1.10:80
```

IP:

```
10.244.1.10
```

Port:

```
80
```

Layer 4 uses both.

---

## Kubernetes Service is Layer 4

Example:

```yaml
kind: Service
spec:
  ports:
  - port: 80
```

Service doesn't care about:

```
/api
/login
/admin
```

It only cares about:

```
TCP 80
TCP 443
UDP 53
```

This is Layer 4 routing.

---

## Layer 4 Load Balancer

Example:

```
Client
   ↓
LB
   ↓
Pod1
Pod2
Pod3
```

LB only sees:

```
TCP Port 80
```

It does NOT understand URLs.

---

# 4. Layer 7 (Application Layer)

Responsible for:

```
HTTP
HTTPS
DNS
Headers
Cookies
URLs
```

Example Request:

```
GET /api/users
```

Only Layer 7 devices understand this.

---

# Ingress is Layer 7

Ingress can inspect:

```
Host Header
Path
HTTP Method
```

Example:

```
app.company.com
```

and

```
/api
```

---

## Example

```
Request:

app.company.com/api
```

Ingress:

```
Route to API Service
```

Request:

```
app.company.com/web
```

Ingress:

```
Route to Web Service
```

Layer 4 cannot do this.

---

# Layer 4 vs Layer 7

| Feature | Layer 4 | Layer 7 |
| --- | --- | --- |
| Understand TCP/UDP | ✅ | ✅ |
| Understand HTTP | ❌ | ✅ |
| Understand URL Path | ❌ | ✅ |
| Understand Hostname | ❌ | ✅ |
| Route by `/api` | ❌ | ✅ |
| Route by domain | ❌ | ✅ |

# Option 1: LoadBalancer Service

```yaml
apiVersion: v1
kind: Service

spec:
  type: LoadBalancer
```

Architecture:

```
Internet
   ↓
Cloud Load Balancer
   ↓
Service
   ↓
Pods
```

---

## Limitation

Suppose:

```
App1
App2
App3
```

You create:

```
LB for App1
LB for App2
LB for App3
```

Result:

```
3 Load Balancers
3 Public IPs
```

Expensive.

---

# Option 2: Ingress

Architecture:

```
Internet
   ↓
Load Balancer
   ↓
Ingress Controller
   ↓
Services
   ↓
Pods
```

---

# Example Ingress

Suppose:

```
api.company.com
```

and

```
shop.company.com
```

---

Ingress YAML:

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress

metadata:
  name: app-ingress

spec:
  rules:

  - host: api.company.com

    http:
      paths:
      - path: /
        pathType: Prefix

        backend:
          service:
            name: api-service
            port:
              number: 80

  - host: shop.company.com

    http:
      paths:
      - path: /
        pathType: Prefix

        backend:
          service:
            name: shop-service
            port:
              number: 80
```

# Why Can't We Use Only Ingress?

This is the most common interview question.

---

## Problem

Ingress is just a Kubernetes object.

Example:

```yaml
kind: Ingress
```

This YAML itself cannot receive internet traffic.

---

You need an actual component:

```
NGINX Ingress Controller
Traefik
HAProxy
```

that listens for traffic.

---

# How Traffic Reaches Ingress?

You expose the Ingress Controller using a LoadBalancer Service.

- **Ingress**: The set of rules for routing traffic.
- **Ingress Controller**: The actual implementation that processes those rules and manages the traffic accordingly.

Example:

```
Internet
   ↓
Load Balancer
   ↓
Ingress Controller
```

Without LB:

```
Internet
      X
Ingress Controller
```

No public entry point.

- You can deploy an Ingress controller without a LoadBalancer by using NodePort or ClusterIP, depending on your requirements. The choice depends on how you want to expose your services and the architecture of your application.

# One-Line Memory Trick

```
Layer 3 → Where should the packet go? (IP)

Layer 4 → Which application port? (TCP/UDP)

Layer 7 → Which website, host, or URL? (HTTP/HTTPS)

LoadBalancer gives external connectivity.

Ingress provides smart HTTP/HTTPS routing.

Ingress usually sits behind a LoadBalancer.
```
