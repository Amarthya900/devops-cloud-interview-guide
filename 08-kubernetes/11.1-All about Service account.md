# Service Account in Kubernetes

## Definition

A **Service Account (SA)** is an identity used by applications (Pods) to authenticate with the Kubernetes API Server.

Think:

| Used By | Identity Type |
| --- | --- |
| Human User | User Account |
| Application/Pod | Service Account |

Example:

```
Developer
   ↓
kubectl get pods

Uses:
User Identity
```

```
Pod
   ↓
Calls Kubernetes API

Uses:
Service Account
```

---

# Why Do We Need Service Accounts?

Suppose your application wants to:

```
Read Pods
Read ConfigMaps
Read Secrets
Create Jobs
Scale Deployments
```

The Kubernetes API Server must know:

```
Who is making this request?
```

That's where Service Accounts come in.

# Step 1: Create Service Account

```yaml
apiVersion: v1
kind: ServiceAccount

metadata:
  name: app-sa
  namespace: dev
```

Apply:

```bash
kubectl apply -f sa.yaml
```

Verify:

```bash
kubectl get sa -n dev
```

Output:

```
NAME
app-sa
```

---

# Step 2: Create Role

Suppose application needs:

```
Get Pods
List Pods
```

Create Role:

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role

metadata:
  name: pod-reader
  namespace: dev

rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs:
  - get
  - list
  - watch
```

---

# What Does This Mean?

```yaml
resources:
- pods
```

Means:

```
Access Pod objects
```

---

```yaml
verbs:
- get
- list
- watch
```

Means:

```
Can view Pods
```

Not create or delete.

---

# Step 3: Create RoleBinding

Now connect:

```
Service Account
      ↓
Role
```

RoleBinding:

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding

metadata:
  name: pod-reader-binding
  namespace: dev

subjects:
- kind: ServiceAccount
  name: app-sa
  namespace: dev

roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
```

Apply:

```bash
kubectl apply -f rolebinding.yaml
```

---

```

   ↓

   ↓
```

# Step 4: Attach Service Account to Pod

```yaml
apiVersion: v1
kind: Pod

metadata:
  name: test-pod

spec:
  serviceAccountName: app-sa

  containers:
  - name: nginx
    image: nginx
```

Notice:

```yaml
serviceAccountName: app-sa
```

This is how the Pod uses that identity.

# Interview Answer

> A Service Account is a Kubernetes identity used by Pods and applications to authenticate with the Kubernetes API Server. It does not grant permissions by itself. Permissions are provided through RBAC objects such as Roles and ClusterRoles, which are associated with the Service Account using RoleBindings or ClusterRoleBindings. The Pod uses the Service Account token to authenticate, and the API Server evaluates RBAC rules to authorize actions.
>

# How Does the Service Account Know About the Role and RoleBinding?

This is the key RBAC concept many people miss.

The Service Account **does not know anything** about Roles or RoleBindings.

Instead, the **Kubernetes API Server** uses the RoleBinding to connect them.

---

# Think Like a Database Relationship

Suppose you have:

```

    app-sa

Role
    pod-reader

RoleBinding
    connects them
```

The relationship is:

```
app-sa
   ↓
RoleBinding
   ↓
pod-reader
```

```yaml
- kind: ServiceAccount
```

Read it literally:

```

  app-sa

  pod-reader
```

Meaning:

```
Give app-sa
the permissions
defined in pod-reader
```

# What Happens When Pod Calls API?

Suppose:

```yaml
spec:
  serviceAccountName: app-sa
```

Pod starts.

Kubernetes mounts:

```
token
```

inside the Pod.

---

Application sends:

```
GET /api/v1/pods
```

to API Server.

Token says:

```
I am app-sa
```

---

API Server checks:

```
Who is app-sa?
```

Then:

```
Any RoleBindings for app-sa?
```

Finds:

```
pod-reader-binding
```

---

Then:

```
Which Role is attached?
```

Finds:

```
pod-reader
```

---

Then:

```
Can pod-reader
list pods?
```

Yes.

Request allowed.

# Important Point

The Service Account itself contains:

```yaml
metadata:
  name: app-sa
```

Only identity.

Nothing else.

There is no field like:

```yaml
role: pod-reader
```

or

```yaml
permissions:
```

inside ServiceAccount.

Permissions are completely external.

# Interview Answer

> A Service Account is linked to permissions indirectly through a RoleBinding or ClusterRoleBinding. The Pod references the Service Account using the `serviceAccountName` field. When the Pod accesses the Kubernetes API, the API Server authenticates it using the Service Account token and then checks RoleBindings to determine which Roles or ClusterRoles are associated with that Service Account before authorizing the request.
> 

## One-Line Memory Trick

```
Pod → ServiceAccount → RoleBinding → Role → Permissions
```

The **only place you explicitly attach the Service Account is in the Pod spec**:

```yaml
serviceAccountName: app-sa
```

Everything else is connected by Kubernetes RBAC behind the scenes.
