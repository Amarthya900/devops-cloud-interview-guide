# 1. How kube-proxy and CoreDNS Work Together

Most beginners memorize:

```
Pod → Service → Pod
```

but don't understand what happens internally.

Let's build it step by step.

---

# Example Setup

## Deployment

```yaml
apiVersion: apps/v1
kind: Deployment

metadata:
  name: nginx

spec:
  replicas: 3

  selector:
    matchLabels:
      app: nginx

  template:
    metadata:
      labels:
        app: nginx

    spec:
      containers:
      - name: nginx
        image: nginx
        ports:
        - containerPort: 80
```

Apply:

```bash
kubectl apply -f deployment.yaml
```

---

## Service

```yaml
apiVersion: v1
kind: Service

metadata:
  name: nginx-service

spec:
  selector:
    app: nginx

  ports:
  - port: 80
    targetPort: 80

  type: ClusterIP
```

Apply:

```bash
kubectl apply -f service.yaml
```

---

# What Happens Internally?

## Step 1: Pods Get IP Addresses

Check:

```bash
kubectl get pods -o wide
```

Output:

```
NAME          IP
nginx-1       10.244.1.10
nginx-2       10.244.1.11
nginx-3       10.244.1.12
```

Each Pod receives its own IP from the cluster network.

---

# Step 2: Service Gets a ClusterIP

Check:

```bash
kubectl get svc
```

Output:

```
NAME            CLUSTER-IP
nginx-service   10.96.0.100
```

Important:

```
10.96.0.100
```

is NOT a Pod.

It is a **virtual IP**.

Think:

```
Frontend Address
```

for the Service.

---

# Step 3: Endpoints Are Created

Check:

```bash
kubectl get endpoints
```

Output:

```
NAME            ENDPOINTS
nginx-service   10.244.1.10:80,10.244.1.11:80,10.244.1.12:80
```

Kubernetes automatically creates this.

Meaning:

```
Service
     ↓
Backend Pods

10.244.1.10
10.244.1.11
10.244.1.12
```

---

# Step 4: kube-proxy Notices the Service

kube-proxy runs on every node.

Check:

```bash
kubectl get pods -n kube-system
```

You'll see kube-proxy Pods.

kube-proxy continuously watches:

```
Services
Endpoints
EndpointSlices
```

through the API Server.

---

# Step 5: kube-proxy Creates Linux Networking Rules

kube-proxy sees:

```
Service:

10.96.0.100

Backends:

10.244.1.10
10.244.1.11
10.244.1.12
```

and programs Linux networking.

Conceptually:

```
IF traffic arrives for

10.96.0.100:80

THEN send to

10.244.1.10:80
10.244.1.11:80
10.244.1.12:80
```

---

# Where Are These Rules Stored?

Usually in:

```
iptables
```

or

```
IPVS
```

depending on kube-proxy mode.

Check mode:

```bash
kubectl logs -n kube-system <kube-proxy-pod>
```

Example:

```
Using iptables Proxier
```

---

# How Traffic Actually Flows

Suppose another Pod sends:

```bash
curl http://10.96.0.100
```

Request flow:

```
Client Pod
     ↓
10.96.0.100
     ↓
iptables/IPVS rule
     ↓
10.244.1.11
     ↓
Nginx Pod
```

Notice:

```
Traffic never stops at kube-proxy.
```

kube-proxy only created the rule.

Linux kernel does the forwarding.

---

# Load Balancing Example

Request 1:

```
10.96.0.100
     ↓
10.244.1.10
```

Request 2:

```
10.96.0.100
     ↓
10.244.1.11
```

Request 3:

```
10.96.0.100
     ↓
10.244.1.12
```

This is Kubernetes Service load balancing.

---

# What Happens If a Pod Dies?

Suppose:

```
10.244.1.11
```

crashes.

New endpoints:

```
10.244.1.10
10.244.1.12
```

Check:

```bash
kubectl get endpoints
```

Output:

```
nginx-service
10.244.1.10:80,10.244.1.12:80
```

kube-proxy notices the change and updates Linux rules automatically.

No manual work needed.

# Where Does DNS Come In?

Without DNS, applications must remember:

```
10.96.0.100
```

which is inconvenient.

Instead applications use:

```
nginx-service
```

---

# CoreDNS

CoreDNS is the DNS server running inside Kubernetes.

Check:

```bash
kubectl get pods -n kube-system
```

Example:

```
coredns-xxxxx
coredns-yyyyy
```

---

# DNS Resolution Flow

Application:

```bash
curl http://nginx-service
```

Question:

```
What is nginx-service?
```

---

# Step 1: DNS Query

Pod sends:

```
Who is nginx-service?
```

to CoreDNS.

---

# Step 2: CoreDNS Checks Kubernetes

CoreDNS watches:

```
Services
Namespaces
EndpointSlices
```

through the API Server.

It finds:

```
nginx-service
     ↓
10.96.0.100
```

---

# Step 3: CoreDNS Responds

```
nginx-service = 10.96.0.100
```

returned to the Pod.

---

# Step 4: Service Routing Starts

Now the Pod sends traffic to:

```
10.96.0.100
```

which is handled by kube-proxy rules.

Flow:

```
Pod
 ↓
CoreDNS
 ↓
10.96.0.100
 ↓
iptables/IPVS
 ↓
Backend Pod
```

---

# Complete End-to-End Flow

```
Application
   |
   | curl nginx-service
   v

CoreDNS
   |
   | resolves
   v

10.96.0.100
(Service IP)

   |
   | kube-proxy created rule
   v

iptables/IPVS

   |
   | load balance
   v

10.244.1.10
10.244.1.11
10.244.1.12

   |
   v

Nginx Pod
```

# Interview Cheat Sheet

| Component | Responsibility |
| --- | --- |
| Deployment | Creates Pods |
| Service | Provides stable virtual IP |
| Endpoints | List of backend Pod IPs |
| kube-proxy | Programs iptables/IPVS rules |
| CoreDNS | Resolves Service names to ClusterIPs |
| iptables/IPVS | Routes traffic to Pods |
| Pod | Serves the application |

### 6-Line Summary

1. Pods receive individual IP addresses from the cluster network.
2. A Service gets a stable virtual IP called a ClusterIP.
3. Kubernetes creates Endpoints containing the IPs of Pods matching the Service selector.
4. kube-proxy watches Services and Endpoints and creates iptables/IPVS rules on every node.
5. Traffic sent to the Service IP is load-balanced to one of the backend Pods.
6. CoreDNS resolves Service names (e.g., `nginx-service`) to the Service ClusterIP so applications can communicate using DNS names instead of IP addresses.

# 2. Kubernetes Networking (Brief Overview)

Yes, the previous explanations covered **Service networking, kube-proxy, CoreDNS, ClusterIP, Endpoints, and Service-to-Pod communication**.

However, Kubernetes Networking is broader than that.

---

# Kubernetes Networking Components

| Component | Purpose |
| --- | --- |
| Pod Networking | Enables Pod-to-Pod communication |
| CNI Plugin | Assigns IPs and configures network interfaces |
| Service Networking | Provides stable virtual IPs for Pods |
| kube-proxy | Routes Service traffic to backend Pods |
| CoreDNS | Provides DNS resolution |
| Ingress | Exposes HTTP/HTTPS applications |
| Network Policies | Control allowed network traffic |
| LoadBalancer | Exposes Services externally |
| NodePort | Exposes Services on node ports |

---

# Kubernetes Networking Model

Kubernetes follows these rules:

### 1. Pod-to-Pod Communication

Every Pod gets its own IP.

```
Pod A (10.244.1.10)
        ↓
Pod B (10.244.2.20)
```

Pods can communicate directly without NAT.

---

### 2. Pod-to-Service Communication

```
Pod
 ↓
Service (ClusterIP)
 ↓
Backend Pods
```

Handled by:

```
kube-proxy
```

---

### 3. External-to-Service Communication

```
Internet
   ↓
LoadBalancer / NodePort / Ingress
   ↓
Service
   ↓
Pods
```

---

# What is a CNI Plugin?

Kubernetes itself does **not** create Pod networking.

It delegates that job to a CNI plugin.

Popular CNIs:

- Calico
- Flannel
- Cilium
- Weave Net

Responsibilities:

```
Assign Pod IPs
Create veth pairs
Configure routes
Enable Pod communication
```

# 30-Second Interview Answer

> Kubernetes networking ensures that every Pod gets a unique IP and can communicate with other Pods across nodes. A CNI plugin such as Calico or Cilium provides Pod networking. Services provide stable virtual IPs, kube-proxy routes Service traffic to backend Pods using iptables or IPVS, and CoreDNS enables DNS-based service discovery. External access is provided through Ingress, NodePort, or LoadBalancer Services.
