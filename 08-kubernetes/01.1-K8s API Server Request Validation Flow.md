# Kubernetes API Server Request Validation Flow (Interview Notes)

## Request Flow

```
User / Component Request
          │
          ▼
1. Authentication
          │
          ▼
2. Authorization
          │
          ▼
3. Admission Controllers
          │
          ▼
4. Schema Validation
          │
          ▼
5. Store in etcd
          │
          ▼
6. Response Back
```

---

# 1. Authentication (Who are you?)

### Purpose

Verifies the identity of the requester.

### Methods

| Method | Example |
| --- | --- |
| Client Certificate | kubelet authenticates using certificate |
| Bearer Token | Service Account token |
| OpenID Connect (OIDC) | Login through Google/Okta |

### Example

```bash
kubectl get pods
```

API Server checks:

> "Is this really user Pi?"
> 

✅ Valid identity → Continue

❌ Invalid identity → Reject request

---

# 2. Authorization (What can you do?)

### Purpose

Checks permissions after identity is verified.

### Common Mechanisms

| Method | Purpose |
| --- | --- |
| RBAC | Most commonly used |
| ABAC | Attribute-based permissions |
| Webhook | External authorization system |

### Example

User wants to delete a pod:

```bash
kubectl delete pod nginx
```

API Server checks:

> "Does this user have delete permission?"
> 

✅ Allowed → Continue

❌ Not Allowed → 403 Forbidden

---

# 3. Admission Controllers (Should I allow this request?)

### Purpose

Apply company policies and cluster rules before storing the object.

Think of them as:

> "Security Guards of Kubernetes"
> 

---

## Position in Request Flow

```
Request
   │
   ▼
Authentication
   │
   ▼
Authorization
   │
   ▼
Admission Controllers
   │
   ▼
etcd
```

Admission Controllers run:

✅ After Authentication

✅ After Authorization

✅ Before etcd

---

## Types

### A. Mutating Admission Controller

Can modify the request.

### Example

Developer creates Pod:

```yaml
resources: {}
```

Admission Controller automatically adds:

```yaml
resources:
  requests:
    cpu: 100m
```

Request changed before saving.

---

### B. Validating Admission Controller

Checks rules but cannot modify.

### Example

Company policy:

> No container should run as root.
> 

Pod submitted:

```yaml
securityContext:
  runAsUser: 0
```

❌ Request rejected.

---

## Common Admission Controllers

| Controller | Purpose |
| --- | --- |
| NamespaceLifecycle | Prevent creation in terminated namespace |
| LimitRanger | Sets/enforces CPU & Memory limits |
| ResourceQuota | Controls namespace resource usage |
| PodSecurity | Enforces pod security standards |
| DefaultStorageClass | Automatically assigns storage class |

---

# 4. Schema Validation (Is YAML Structure Correct?)

### Purpose

Checks whether the object follows Kubernetes API schema.

### Example

Correct:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx
```

Wrong:

```yaml
apiVer: v1
```

❌ Invalid field.

API Server rejects request.

---

## Checks Performed

| Validation | Example |
| --- | --- |
| Required fields | apiVersion, kind |
| Data types | replicas must be integer |
| Supported API version | apps/v1 |
| Resource format | Valid YAML/JSON |

---

# 5. Persistence in etcd

After all checks pass:

```
Authentication ✓
Authorization ✓
Admission Controller ✓
Schema Validation ✓
```

Object stored in:

```
etcd
```

Cluster state becomes official.

---

# 6. Rate Limiting

### Purpose

Protect API Server from overload.

### Example

A script sends:

```bash
10000 requests/sec
```

API Server limits requests.

Benefits:

- Prevents abuse
- Protects cluster stability
- Ensures fair usage

---

# Admission Controllers Implementation

## Where?

Inside the API Server.

Configured using:

```bash
kube-apiserver \
--enable-admission-plugins=NamespaceLifecycle,LimitRanger,ResourceQuota
```

---

## How?

### Built-in Controllers

Simply enable them:

```bash
--enable-admission-plugins
```

Example:

- LimitRanger
- ResourceQuota
- PodSecurity

---

### Custom Admission Controllers

Usually implemented as:

```
Mutating Webhook
or
Validating Webhook
```

Steps:

1. Write webhook application (Go/Python/Java)
2. Deploy in Kubernetes
3. Expose via Service
4. Register webhook with API Server
5. API Server calls webhook during admission phase

---

## Real-World Example

Company Rule:

> Every pod must contain team label.
> 

Developer creates:

```yaml
metadata:
  name: nginx
```

Webhook checks:

```yaml
labels:
  team: devops
```

If missing:

❌ Reject Request

If present:

✅ Allow Request

# Interview Answer (30 Seconds)

**How does Kubernetes API Server validate requests?**

1. Authentication → verifies identity.
2. Authorization → checks permissions using RBAC.
3. Admission Controllers → enforce policies and may modify requests.
4. Schema Validation → validates YAML/JSON structure.
5. Object is stored in etcd.
6. Rate limiting protects the API Server from excessive requests.
