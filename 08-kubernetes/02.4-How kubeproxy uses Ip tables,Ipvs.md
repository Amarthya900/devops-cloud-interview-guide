# What Exactly is IPVS?

You already know:

```
Service IP
    ↓
kube-proxy
    ↓
Pod IP
```

The question is:

```
How does kube-proxy make Linux perform
this routing?
```

There are two common methods:

| Mode | Technology |
| --- | --- |
| iptables | Linux firewall/NAT rules |
| IPVS | Linux Layer-4 load balancer |

---

# What is IPVS?

**IPVS = IP Virtual Server**

It is a Linux kernel feature that acts as a:

```
Layer 4 Load Balancer
```

It can load balance:

```
TCP
UDP
SCTP
```

traffic among multiple backend servers.

Think:

```
AWS ELB
NGINX
HAProxy
F5
```

but implemented directly inside the Linux kernel.

---

# Before Kubernetes Existed

People already used IPVS for:

```
Load Balancers
Web Farms
High Availability Systems
```

Example:

```
VIP: 10.0.0.100

Backend Servers:

10.0.0.11
10.0.0.12
10.0.0.13
```

IPVS automatically distributed traffic.

---

# How kube-proxy Uses IPVS

Suppose:

```
Service IP

10.96.0.100
```

Backend Pods:

```
10.244.1.10
10.244.1.11
10.244.1.12
```

kube-proxy programs IPVS like:

```
10.96.0.100:80

      ↓

10.244.1.10:80
10.244.1.11:80
10.244.1.12:80
```

Then Linux kernel performs load balancing.

kube-proxy does NOT forward packets itself.

---

# Why Not Use iptables?

iptables works.

But imagine:

```
10 Services
```

No problem.

---

Now imagine:

```
500 Services
5000 Pods
```

iptables becomes huge.

Linux must evaluate many rules.

Performance degrades.

---

IPVS stores information in efficient kernel data structures:

```
Hash Tables
Connection Tables
```

Lookup becomes much faster.

---

```bash
kubectl -n kube-system logs kube-proxy-xxxxx
```

Look for:

```

```

or

```
Using ipvs Proxier
```

# How To See IPVS Rules

Install:

```bash
sudo apt install ipvsadm
```

Check:

```bash
sudo ipvsadm -Ln
```

Example:

```
TCP 10.96.0.100:80

 -> 10.244.1.10:80
 -> 10.244.1.11:80
 -> 10.244.1.12:80
```

This is the actual Service.

Not Kubernetes abstraction.

Actual Linux kernel entries.

---

# How To See iptables Rules

Check:

```bash
sudo iptables -t nat -L
```

You may see:

```
KUBE-SERVICES

KUBE-SVC-xxxxx

KUBE-SEP-xxxxx
```

These chains are created by kube-proxy.

# What You Should Remember for Interviews

```
kube-proxy does not route traffic itself.

It programs Linux networking.

iptables:
    Uses firewall/NAT rules.

IPVS:
    Uses Linux kernel load balancer.

Both ultimately route Service traffic
to backend Pod IPs.
```

---

# Mental Model

Think of kube-proxy as:

```
Network Administrator
```

It tells Linux:

```
"If traffic comes to
Service IP 10.96.0.100,

send it to one of these Pods:

10.244.1.10
10.244.1.11
10.244.1.12"
```

Then:

```
iptables or IPVS
does the actual work.
```

So **IPVS is not an abstraction**. It is a real Linux kernel subsystem that you can inspect (`ipvsadm`), troubleshoot, and interact with as a DevOps/SRE engineer when diagnosing Kubernetes networking issues.

# 2. What are tcpdump, ss, netstat, and conntrack? Why Does an SRE Use Them?

Don't think of these as Kubernetes tools.

Think of them as **Linux networking troubleshooting tools**.

When Kubernetes networking breaks, an SRE often works from:

```
Kubernetes
    ↓
kube-proxy
    ↓
Linux Networking
    ↓
Kernel
```

Eventually you have to inspect what Linux is actually doing.

---

# Real Scenario

Suppose:

```
frontend Pod
     ↓
Service
     ↓
backend Pod
```

Frontend cannot reach backend.

Users complain:

```
Application Timeout
```

Now the SRE starts investigating.

---

# Tool #1: ss

## What is it?

Shows:

```
Open Network Connections
Listening Ports
TCP Sessions
```

Think:

```
Which processes are talking
to which IPs and ports?
```

---

### Example

```bash
ss -tulpn
```

Output:

```
LISTEN
0.0.0.0:80
0.0.0.0:443
0.0.0.0:10250
```

Meaning:

```
Port 80 is open
Port 443 is open
Port 10250 is open
```

---

### SRE Use Case

Suppose Nginx Pod should listen on:

```
8080
```

Check:

```bash
ss -tulpn
```

Output:

```
LISTEN 8080
```

Good.

If missing:

```
Application isn't listening.
```

Problem found.

---

# Tool #2: netstat

Older version of `ss`.

Historically everyone used:

```bash
netstat -tulpn
```

Nowadays:

```bash
ss
```

is preferred because it's faster.

Interview answer:

```
ss is the modern replacement
for netstat.
```

---

# Tool #3: tcpdump

This is the most important one.

## What is it?

Shows actual packets moving on the network.

Think:

```
Wireshark
but in terminal
```

---

Example:

```bash
tcpdump -i any port 80
```

Now Linux prints:

```
Packet received
Packet sent
Packet received
```

---

## Why SRE Uses It

Suppose:

```
Frontend
   ↓
Service
   ↓
Backend
```

fails.

Question:

```
Is traffic reaching backend?
```

Run:

```bash
tcpdump -i any port 8080
```

---

### Case 1

You see packets:

```
GET /api
GET /api
GET /api
```

Meaning:

```
Traffic arrived.
```

Network is fine.

Application is the problem.

---

### Case 2

No packets appear.

Meaning:

```
Traffic never reached backend.
```

Problem is somewhere in networking.

Huge clue.

---

# Tool #4: conntrack

Most confusing one.

---

## What is a Connection?

Suppose:

```
Pod A
    ↓
Pod B
```

TCP connection created.

Linux keeps track:

```
Source IP
Destination IP
Port
State
```

in a table called:

```
Connection Tracking Table
```

or

```
conntrack
```

---

Example Entry

```
10.244.1.10:50000
        ↓
10.244.2.20:80

ESTABLISHED
```

Linux remembers:

```
These two are talking.
```

---

## Why SRE Uses It

Suppose:

```
Service randomly fails.
```

Check:

```bash
conntrack -L
```

You may discover:

```
Connection table full
```

or

```
Thousands of stale connections
```

Then:

```
Packets get dropped.
```

---

# How This Relates to Kubernetes

Let's say:

```
frontend Pod
    ↓
nginx-service
    ↓
backend Pod
```

Request fails.

---

SRE Investigation

### Step 1

Check Service:

```bash
kubectl get svc
```

---

### Step 2

Check Endpoints:

```bash
kubectl get endpoints
```

Output:

```
10.244.1.10
10.244.1.11
```

Looks good.

---

### Step 3

Check Pod

```bash
kubectl exec
```

Can backend start?

---

### Step 4

Check open ports

```bash
ss -tulpn
```

Verify application is listening.

---

### Step 5

Check packets

```bash
tcpdump
```

Question:

```
Did packets arrive?
```

---

### Step 6

Check connection tracking

```bash
conntrack -L
```

Question:

```
Is Linux dropping connections?
```

# What a Junior DevOps Engineer Should Know

You do **not** need to memorize commands.

Understand the purpose:

| Tool | What Question Does It Answer? |
| --- | --- |
| ss | Is my application listening on the port? |
| netstat | Same as ss (older) |
| tcpdump | Are packets actually reaching this machine/pod? |
| conntrack | Is Linux tracking/dropping connections correctly? |

---

# Interview One-Liner

> `ss` checks listening ports and active connections, `tcpdump` captures network packets, `conntrack` inspects Linux connection-tracking state, and SREs use these tools to troubleshoot Kubernetes networking issues when traffic is not reaching Services or Pods.
