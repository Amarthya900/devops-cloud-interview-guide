# Kubernetes Deployment vs ReplicaSet - Complete Notes

## 1. What Problem Do They Solve?

Kubernetes applications run inside Pods.

Pods can crash, nodes can fail, or containers can exit unexpectedly.

Kubernetes uses controllers to ensure the desired state is maintained.

Two important controllers are:

| Controller | Responsibility                              |
| ---------- | ------------------------------------------- |
| ReplicaSet | Maintains the desired number of Pods        |
| Deployment | Manages ReplicaSets and application updates |

---

# 2. ReplicaSet

A ReplicaSet ensures a specified number of Pod replicas are always running.

Example:

```yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: nginx-rs

spec:
  replicas: 3

  selector:
    matchLabels:
      app: nginx

  template:
    metadata:
      labels:
        app: nginx

    spec:
      containers:
      - name: nginx
        image: nginx
```

### What Happens?

```text
ReplicaSet
    ↓
3 Pods
```

If one Pod dies:

```text
Desired Pods = 3
Current Pods = 2
```

ReplicaSet automatically creates another Pod.

Result:

```text
Always 3 Pods running
```

---

# 3. Limitations of ReplicaSet

ReplicaSet only knows:

```text
Maintain Pod Count
```

It does NOT know:

```text
Rolling Updates
Rollbacks
Version History
Deployment Strategies
```

Suppose:

```text
nginx:1.25 → nginx:1.26
```

A ReplicaSet alone cannot safely update all Pods.

You would need to manually create another ReplicaSet and manage the transition.

---

# 4. Deployment

A Deployment is a higher-level controller.

Instead of managing Pods directly, it manages ReplicaSets.

Actual hierarchy:

```text
Deployment
     ↓
ReplicaSet
     ↓
Pods
```

Example:

```yaml
apiVersion: apps/v1
kind: Deployment

metadata:
  name: nginx-deployment

spec:
  replicas: 3

  selector:
    matchLabels:
      app: nginx

  template:
    metadata:
      labels:
        app: nginx

    spec:
      containers:
      - name: nginx
        image: nginx:1.25

        ports:
        - containerPort: 80
```

Apply:

```bash
kubectl apply -f deployment.yaml
```

---

# 5. What Happens Internally?

### Step 1

Deployment object gets created.

```text
Deployment
nginx-deployment
```

---

### Step 2

Deployment automatically creates a ReplicaSet.

```text
nginx-deployment-7d4f8c9b7
```

Now:

```text
Deployment
      ↓
ReplicaSet
```

---

### Step 3

ReplicaSet creates Pods.

```text
Deployment
      ↓
ReplicaSet
      ↓
3 Pods
```

Example:

```text
nginx-deployment-7d4f8c9b7-x1
nginx-deployment-7d4f8c9b7-x2
nginx-deployment-7d4f8c9b7-x3
```

Final structure:

```text
Deployment
      ↓
ReplicaSet
      ↓
Pods
      ↓
Containers
```

---

# 6. Understanding the ReplicaSet Name

Example:

```text
nginx-deployment-7d4f8c9b7
```

Breakdown:

```text
nginx-deployment
       +
Template Hash
       =
ReplicaSet Name
```

Where:

```text
7d4f8c9b7
```

is a hash generated from the Pod template.

This helps Kubernetes identify different application versions.

---

# 7. Rolling Update Example

Current Deployment:

```yaml
image: nginx:1.25
```

Structure:

```text
Deployment
     ↓
ReplicaSet-v1
     ↓
3 Pods (1.25)
```

Update:

```yaml
image: nginx:1.26
```

Apply again:

```bash
kubectl apply -f deployment.yaml
```

Deployment detects:

```text
Pod Template Changed
```

and creates a new ReplicaSet.

```text
Deployment
      ↓

ReplicaSet-v1
(image 1.25)

ReplicaSet-v2
(image 1.26)
```

Then Kubernetes performs:

```text
Scale Up ReplicaSet-v2
Scale Down ReplicaSet-v1
```

Result:

```text
Deployment
      ↓
ReplicaSet-v2
      ↓
3 Pods (1.26)
```

This process is called:

```text
Rolling Update
```

---

# 8. Rollback Example

Suppose:

```text
v1.25 = Working
v1.26 = Broken
```

Rollback:

```bash
kubectl rollout undo deployment nginx-deployment
```

Deployment uses the previous ReplicaSet.

```text
Current ReplicaSet → Scale Down

Previous ReplicaSet → Scale Up
```

Result:

```text
Back to nginx:1.25
```

# 9. How to Read a Deployment YAML

## Structure

```yaml
apiVersion:
kind:
metadata:
spec:
```

Think:

| Section    | Meaning                  |
| ---------- | ------------------------ |
| apiVersion | Which API version to use |
| kind       | What object to create    |
| metadata   | Object identity          |
| spec       | Desired state            |

---

## apiVersion

```yaml
apiVersion: apps/v1
```

Meaning:

```text
Use Deployment APIs from apps/v1
```

---

## kind

```yaml
kind: Deployment
```

Meaning:

```text
Create a Deployment object
```

---

## metadata

```yaml
metadata:
  name: nginx-deployment
```

Meaning:

```text
Object Name = nginx-deployment
```

---

## spec

```yaml
spec:
```

Think:

```text
Desired State
```

Everything inside `spec` describes what Kubernetes should maintain.

---

## replicas

```yaml
replicas: 3
```

Meaning:

```text
Always keep 3 Pods running
```

---

## selector

```yaml
selector:
  matchLabels:
    app: nginx
```

Meaning:

```text
Manage Pods having:

app=nginx
```

---

## template

```yaml
template:
```

Think:

```text
Blueprint for future Pods
```

Whenever a Pod is created, Kubernetes uses this template.

---

## template.metadata.labels

```yaml
labels:
  app: nginx
```

Every Pod receives:

```text
app=nginx
```

---

## template.spec

```yaml
template:
  spec:
```

This is the Pod specification.

Important:

```text
Deployment Spec
     ↓
Pod Template
     ↓
Pod Spec
```

---

## containers

```yaml
containers:
```

List of containers inside the Pod.

---

## image

```yaml
image: nginx:1.25
```

Container image to run.

---

## containerPort

```yaml
containerPort: 80
```

Container listens on port 80.

Note:

```text
containerPort does NOT expose the application.
```

For external access you need a Service.

# 10. Why Does ReplicaSet YAML Exist If Deployment Creates ReplicaSets?

This is a very common interview question.

### Short Answer

Because ReplicaSet is a standalone Kubernetes object.

You can create it directly if you want.

Example:

```yaml
kind: ReplicaSet
```

or

```yaml
kind: Deployment
```

Both are valid.

---

## Then Why Use Deployments?

Because Deployments provide:

| Feature               | ReplicaSet | Deployment |
| --------------------- | ---------- | ---------- |
| Maintain Pod Count    | ✅          | ✅          |
| Rolling Updates       | ❌          | ✅          |
| Rollbacks             | ❌          | ✅          |
| Revision History      | ❌          | ✅          |
| Deployment Strategies | ❌          | ✅          |
| Production Ready      | ❌          | ✅          |

In real production environments:

```text
Almost always use Deployments
```

---

## When Would You Create a ReplicaSet Directly?

Mostly:

### 1. Learning Kubernetes

To understand how ReplicaSets work internally.

### 2. Certification Exams

Questions may ask about ReplicaSets.

### 3. Rare Low-Level Control Scenarios

When you intentionally want Pod replication without Deployment features.

These situations are uncommon.

---

# Interview Answer

**Q: Why don't we create ReplicaSets directly if Deployments create them automatically?**

**Answer:**

> A Deployment is a higher-level controller that manages ReplicaSets. While ReplicaSets can be created directly, they only maintain the desired number of Pods. Deployments add rolling updates, rollbacks, revision history, and deployment strategies. Therefore, in production environments, Deployments are preferred, and ReplicaSets are usually created automatically by Deployments.
