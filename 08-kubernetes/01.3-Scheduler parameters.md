# Index

| Scheduling Factor | Purpose | Example |
| --- | --- | --- |
| **Resource Requests & Limits** | Scheduler checks whether a node has sufficient CPU, memory, and other resources before placing a Pod. | Pod requests `1 CPU` and `2Gi RAM`; scheduler chooses only nodes with at least those available resources. |
| **Node Affinity** | Schedule Pods onto specific nodes based on node labels. | Run database Pods only on nodes labeled `disktype=ssd`. |
| **Pod Affinity** | Schedule Pods close to other Pods that match specific labels. | Place application Pods on the same node as Redis Pods to reduce network latency. |
| **Pod Anti-Affinity** | Prevent Pods from being scheduled near certain Pods, typically to improve availability. | Ensure Nginx replicas run on different nodes. |
| **Taints** | Prevent Pods from being scheduled onto specific nodes unless explicitly allowed. | Reserve a node for databases using `dedicated=db:NoSchedule`. |
| **Tolerations** | Allow Pods to be scheduled onto nodes that have matching taints. | Database Pods tolerate the `dedicated=db:NoSchedule` taint and can run on DB nodes. |
| **Topology Spread Constraints** | Distribute Pods evenly across nodes, zones, or topology domains for high availability. | Spread 6 replicas evenly across 3 nodes (`2-2-2`) or 3 availability zones. |

# Where Do We Write Taints, Affinities, Resources, and Scheduler Rules?

This is the key concept:

**You do NOT write scheduler rules in a separate Scheduler YAML.**

You write them inside the **Pod specification (`spec.template.spec`)** of a Deployment, ReplicaSet, StatefulSet, Job, or Pod.

The scheduler reads these rules from the Pod spec and decides where to place the Pod.

---

# Scheduler Workflow

When you create a Deployment:

```yaml
kind: Deployment
```

Kubernetes does:

```
Deployment
    ↓
ReplicaSet
    ↓
Pod Created (Pending)
    ↓
Scheduler Reads Pod Spec
    ↓
Checks Rules
    ↓
Selects Node
    ↓
Updates Pod.nodeName
    ↓
Pod Runs
```

Inside:

```yaml

  containers:
```

Example:

```yaml
spec:

    image: nginx

    resources:

        cpu: "2"

```

Suppose:

```
worker1
Available:
CPU=2
RAM=4Gi
```

```

RAM=1Gi
```

```
Eligible
```

---

Suppose:

```
worker2
Available:
CPU=500m
RAM=512Mi
```

Pod requests:

```
CPU=1
RAM=1Gi
```


# 2. Node Affinity

## Goal

Choose specific nodes.

Example:

```
Run only on SSD nodes
```

---

### Node Label

```bash
kubectl label node worker1 disktype=ssd
```

---

### YAML

```yaml
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: disktype
            operator: In
            values:
            - ssd
```

---

### Scheduler Logic

```
worker1 → ssd → Eligible
worker2 → hdd → Rejected
worker3 → ssd → Eligible
```

Result:

```
worker1 or worker3
```

---

# 5. Taints

## Goal

Prevent Pods from entering a node.

Node says:

```
Stay Away
```

---

### Apply Taint

```bash
kubectl taint nodes worker1 dedicated=db:NoSchedule
```

Node becomes:

```
worker1
dedicated=db
NoSchedule
```

---

### Scheduler Logic

Normal Pod:

```yaml
spec:
  containers:
  - image: nginx
```

Scheduler:

```
worker1 → Rejected
worker2 → Eligible
worker3 → Eligible
```

# 6. Tolerations

## Goal

Allow Pod onto tainted node.

Pod says:

```
I tolerate this taint
```

---

### YAML

```yaml
spec:
  tolerations:
  - key: "dedicated"
    operator: "Equal"
    value: "db"
    effect: "NoSchedule"
```

---

### Scheduler Logic

Without toleration:

```
worker1 → Rejected
```

With toleration:

```
worker1 → Eligible
```

# 7. Topology Spread Constraints (Ref below for more info)

## Goal

Distribute Pods evenly.

---

### YAML

```yaml
spec:
  topologySpreadConstraints:

  - maxSkew: 1

    topologyKey: kubernetes.io/hostname

    whenUnsatisfiable: DoNotSchedule

    labelSelector:
      matchLabels:
        app: nginx
```

---

### Scheduler Result

Bad:

```
worker1 → 6 pods
worker2 → 0
worker3 → 0
```

Good:

```
worker1 → 2
worker2 → 2
worker3 → 2
```

# Complete Deployment Example

This single Deployment uses:

- Resource Requests
- Node Affinity
- Tolerations
- Pod Anti-Affinity

```yaml
apiVersion: apps/v1
kind: Deployment

metadata:
  name: nginx

spec:
  replicas: 3

  selector:
    matchLabels:
      app: nginx

  template:

    metadata:
      labels:
        app: nginx

    spec:

      tolerations:
      - key: "dedicated"
        operator: "Equal"
        value: "app"
        effect: "NoSchedule"

      affinity:

        nodeAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
            nodeSelectorTerms:
            - matchExpressions:
              - key: disktype
                operator: In
                values:
                - ssd

        podAntiAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
          - labelSelector:
              matchExpressions:
              - key: app
                operator: In
                values:
                - nginx

            topologyKey: kubernetes.io/hostname

      containers:
      - name: nginx
        image: nginx

        resources:
          requests:
            cpu: "500m"
            memory: "512Mi"

          limits:
            cpu: "1"
            memory: "1Gi"
```

# How Scheduler Chooses a Node

For every Pending Pod:

| Step | Check |
| --- | --- |
| 1 | Taints/Tolerations |
| 2 | Node Affinity |
| 3 | Resource Requests |
| 4 | Pod Affinity |
| 5 | Pod Anti-Affinity |
| 6 | Topology Spread |
| 7 | Score Remaining Nodes |
| 8 | Pick Best Node |
| 9 | Update `nodeName` |

# What Does Scheduler Actually Update?

Before scheduling:

```yaml
spec:
  containers:
  - image: nginx
```

Pod status:

```
Pending
```

After scheduler selects worker2:

```yaml
spec:
  nodeName: worker2

  containers:
  - image: nginx
```

Scheduler writes:

```yaml
nodeName: worker2
```

Then kubelet on worker2 starts the container.

# Explanation of Kubernetes Topology Spread Constraints

## Problem Without Topology Spread

Imagine a Deployment with 6 replicas:

```yaml
replicas: 6
```

Cluster:

```
worker1
worker2
worker3
```

Without any spreading rules, the scheduler might place pods like:

```
worker1
 ├─ pod1
 ├─ pod2
 ├─ pod3
 ├─ pod4
 ├─ pod5
 └─ pod6

worker2
 └─ No Pods

worker3
 └─ No Pods
```

### Problem

If `worker1` crashes:

```
6 Pods Lost
Application Down
```

This is called a **single point of failure**.

---

# Solution: Topology Spread Constraints

Tell Kubernetes:

> "Spread my Pods evenly across nodes or zones."
> 

---

# Example YAML

```yaml
apiVersion: apps/v1
kind: Deployment

metadata:
  name: nginx

spec:
  replicas: 6

  selector:
    matchLabels:
      app: nginx

  template:
    metadata:
      labels:
        app: nginx

    spec:

      topologySpreadConstraints:

      - maxSkew: 1

        topologyKey: kubernetes.io/hostname

        whenUnsatisfiable: DoNotSchedule

        labelSelector:
          matchLabels:
            app: nginx

      containers:
      - name: nginx
        image: nginx
```

---

# Understanding Every Field

## 1. labelSelector

```yaml
labelSelector:
  matchLabels:
    app: nginx
```

Means:

```
Apply spreading rules
to nginx Pods only
```

Scheduler counts only Pods having:

```
app=nginx
```

---

## 2. topologyKey

```yaml
topologyKey: kubernetes.io/hostname
```

Means:

```
Spread across nodes
```

Because:

```
Each hostname = one node
```

Example:

```
worker1
worker2
worker3
```

---

### Other Common Values

Spread across Zones:

```yaml
topologyKey: topology.kubernetes.io/zone
```

Example:

```
Zone-A
Zone-B
Zone-C
```

---

## 3. maxSkew

```yaml
maxSkew: 1
```

Means:

```
Difference between
most-loaded and
least-loaded topology
must not exceed 1
```

---

### Valid Distribution

6 Pods

```
worker1 → 2 Pods
worker2 → 2 Pods
worker3 → 2 Pods
```

Difference:

```
2 - 2 = 0
```

Allowed ✅

---

Another valid distribution:

```
worker1 → 3 Pods
worker2 → 2 Pods
worker3 → 1 Pod
```

Difference:

```
3 - 1 = 2
```

Not Allowed ❌

Because:

```
maxSkew = 1
```

---

Valid Example:

```
worker1 → 2
worker2 → 2
worker3 → 1
```

Difference:

```
2 - 1 = 1
```

Allowed ✅

---

## 4. whenUnsatisfiable

```yaml
whenUnsatisfiable: DoNotSchedule
```

Means:

```
If spreading rule
cannot be satisfied,
don't schedule Pod
```

---

Example

Current:

```
worker1 → 3 Pods
worker2 → 2 Pods
worker3 → 2 Pods
```

Now worker2 and worker3 become unavailable.

Only:

```
worker1
```

is available.

Scheduler thinks:

```
Adding another Pod
would violate maxSkew=1
```

Result:

```
Pod stays Pending
```

Because:

```yaml
whenUnsatisfiable: DoNotSchedule
```

# Node vs Zone Spreading

## Node Spreading

```yaml
topologyKey: kubernetes.io/hostname
```

Result:

```
worker1 → 2 Pods
worker2 → 2 Pods
worker3 → 2 Pods
```

Protects against:

```
Node Failure
```

---

## Zone Spreading

```yaml
topologyKey: topology.kubernetes.io/zone
```

Result:

```
Zone-A → 2 Pods
Zone-B → 2 Pods
Zone-C → 2 Pods
```

Protects against:

```
Entire Availability Zone Failure
```

# Difference from Pod Anti-Affinity

## Pod Anti-Affinity

```
Never place matching Pods
on the same node
```

Example:

```
worker1 → pod1
worker2 → pod2
worker3 → pod3
```

Strict separation.

---

## Topology Spread

```
Spread Pods as evenly
as possible
```

Example:

```
worker1 → 2 Pods
worker2 → 2 Pods
worker3 → 2 Pods
```

Balanced distribution.

# Interview Answer (6 Lines)

**Topology Spread Constraints** ensure Pods are distributed evenly across nodes or zones to improve availability and fault tolerance. The scheduler uses `topologyKey` to determine the spreading domain (node or zone). `labelSelector` identifies which Pods should be counted. `maxSkew` controls the maximum allowed difference in Pod count between domains. `whenUnsatisfiable` defines whether Pods should remain Pending or be scheduled anyway when the rule cannot be met. It helps prevent all replicas from being placed on a single node or availability zone.
