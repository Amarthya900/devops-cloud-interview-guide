# Kubernetes Scheduler Jobs – Consolidated Notes

# One-Line Memory Trick

```text
Node Affinity       → Choose WHERE to run
Pod Affinity        → Run TOGETHER
Pod Anti-Affinity   → Run APART
Taints              → KEEP OUT
Tolerations         → ALLOW ENTRY
Resources           → Enough CPU/RAM?
Topology Spread     → Balance replicas
```

The **Kubernetes Scheduler** is responsible for deciding **which node a Pod should run on**.

### Scheduler Workflow

```text
Pod Created
    ↓
Scheduler Watches Pending Pods
    ↓
Filters Suitable Nodes
    ↓
Scores Remaining Nodes
    ↓
Selects Best Node
    ↓
Pod Assigned to Node
```

During scheduling, Kubernetes evaluates several factors:

| Factor                      | Purpose                                  |
| --------------------------- | ---------------------------------------- |
| Node Affinity               | Run pod on preferred/specific nodes      |
| Pod Affinity                | Run pod near other pods                  |
| Pod Anti-Affinity           | Avoid running near certain pods          |
| Taints & Tolerations        | Keep pods away unless explicitly allowed |
| Resource Availability       | Ensure CPU/Memory are available          |
| Topology Spread Constraints | Distribute pods evenly                   |

# 1. Node Affinity

## What is it?

Allows pods to choose nodes based on node labels.

Think:

```text
"I want my pod to run only on SSD nodes."
```

### Label a Node

```bash
kubectl label node worker1 disktype=ssd
```

### Example

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: disktype
            operator: In
            values:
            - ssd
  containers:
  - name: nginx
    image: nginx
```

### Meaning

```text
Only schedule pod on nodes
having label:

disktype=ssd
```

### Interview Question

Why use Node Affinity instead of nodeSelector?

| nodeSelector     | Node Affinity      |
| ---------------- | ------------------ |
| Simple           | Advanced           |
| Exact match only | Multiple operators |
| Less flexible    | Highly flexible    |

# 2. Pod Affinity

## What is it?

Schedule a pod close to another pod.

Think:

```text
Web Pod wants to be near Cache Pod
for lower latency.
```

### Example

```yaml
affinity:
  podAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
    - labelSelector:
        matchExpressions:
        - key: app
          operator: In
          values:
          - redis
      topologyKey: kubernetes.io/hostname
```

### Meaning

```text
Place this pod on the same node
where Redis pod is running.
```

### Real World Example

```text
Frontend
    ↓
Redis Cache

Keep them together
for faster communication.
```

# 3. Pod Anti-Affinity

## What is it?

Prevent pods from running together.

Think:

```text
Don't place two replicas
on the same node.
```

### Example

```yaml
affinity:
  podAntiAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
    - labelSelector:
        matchExpressions:
        - key: app
          operator: In
          values:
          - nginx
      topologyKey: kubernetes.io/hostname
```

### Meaning

```text
If one nginx pod exists on node1,
schedule next nginx pod elsewhere.
```

### Real World Example

```text
Node1
 └─ nginx-1

Node2
 └─ nginx-2

Node3
 └─ nginx-3
```

### Benefit

Higher Availability

If Node1 crashes:

```text
Only nginx-1 lost
Others continue serving traffic
```

# 4. Taints and Tolerations

## What is it?

Node says:

```text
"Do not schedule pods here
unless they tolerate me."
```

### Taint a Node

```bash
kubectl taint nodes worker1 dedicated=db:NoSchedule
```

### Meaning

```text
Keep all pods away from worker1
```

### Toleration Example

```yaml
tolerations:
- key: "dedicated"
  operator: "Equal"
  value: "db"
  effect: "NoSchedule"
```

### Meaning

```text
This pod is allowed
to run on worker1
```

### Real World Example

Dedicated Database Node

```text
worker1
 └─ PostgreSQL

worker2
 └─ Application Pods

worker3
 └─ Application Pods
```

Only DB pods can enter worker1.

# 5. Resource Availability

## What is it?

Scheduler checks:

```text
CPU
Memory
Ephemeral Storage
```

before placing a pod.

### Example

Node:

```text
4 CPU
8 GB RAM
```

Pod requests:

```yaml
resources:
  requests:
    cpu: "2"
    memory: "4Gi"
```

### Scenario

```text
Node Available:
1 CPU
2 GB RAM

Pod Requests:
2 CPU
4 GB RAM
```

Result:

```text
Scheduling Failed
```

Pod remains:

```text
Pending
```

### Check

```bash
kubectl describe pod pod-name
```

Output:

```text
0/3 nodes available:
Insufficient cpu
```

# 6. Topology Spread Constraints

## What is it?

Ensures pods are distributed evenly.

Think:

```text
Don't put all replicas
in one zone/node.
```

### Example

```yaml
topologySpreadConstraints:
- maxSkew: 1
  topologyKey: kubernetes.io/hostname
  whenUnsatisfiable: DoNotSchedule
  labelSelector:
    matchLabels:
      app: nginx
```

### Meaning

For 6 replicas:

```text
Node1 -> 2 pods
Node2 -> 2 pods
Node3 -> 2 pods
```

instead of:

```text
Node1 -> 6 pods
Node2 -> 0
Node3 -> 0
```

### Real World Example

Multi-AZ Deployment

```text
Zone-A -> 2 Pods
Zone-B -> 2 Pods
Zone-C -> 2 Pods
```

If one zone fails:

```text
Application still works
```

# Complete Scheduling Example

Imagine an e-commerce application.

### Requirements

```text
1. Run only on SSD nodes
2. Stay near Redis
3. Avoid same-node replicas
4. Allow dedicated app nodes
5. Need 2 CPU, 4GB RAM

```

```text

Node Affinity

Find SSD nodes

Taints/Tolerations
↓
Check allowed nodes

Step 3:
Resources
↓
Enough CPU/RAM?

Step 4:
Pod Affinity
↓
Near Redis?

Step 5:
Pod Anti-Affinity
↓
Avoid same-node replicas

Step 6:
Topology Spread
↓
Balance across zones

Step 7:
Choose Best Node
```
